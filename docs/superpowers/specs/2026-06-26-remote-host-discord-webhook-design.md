# Remote Host Support via Discord Webhook

**Date:** 2026-06-26
**Status:** TODO

## Problem

Cogito's current security model requires a LAN-realm player to act as host — only they can start games, reset sessions, and configure AI slots. This works when the server operator is home, but breaks entirely when everyone (including the operator) is remote. The goal is to allow a fully remote group to play while keeping the same access-control guarantees: uninvited strangers can't join, and only a trusted person controls the session.

## Design

Replace the hard LAN-realm requirement for host privileges with a **rotating host secret**. Both the host secret and the player join code are pre-generated and delivered to a configured Discord webhook on every rotation event. The operator forwards the host secret to a designated friend; that friend presents it in the existing `code` field at join time and receives host privileges. Everyone else joins with the regular join code as before.

The LAN realm path is preserved unchanged — if the operator is home, they join as a LAN player with no code required.

## User Flow

1. Server starts → generates host secret + join code → POSTs both to Discord
2. Operator sees Discord message, DMs designated friend the host secret
3. Operator pastes join code to group chat
4. Friend enters name + host secret → becomes host, can configure and start game
5. Everyone else enters name + join code → joins as regular player

## Rotation Events

| Trigger | Discord reason label |
|---|---|
| Server startup | `startup` |
| `lobby:reset` | `session reset` |
| `game:returnToLobby` | `session reset` |
| Daily at midnight (server time) | `daily rotation` |

Daily rotation fires unconditionally — mid-game rotation is harmless since joining is blocked during gameplay. Implementation: on startup, compute ms until next midnight and schedule a one-shot timer; on fire, rotate then schedule the next midnight.

## Discord Message Format

```
🎮 Cogito session codes rotated [startup]
Host secret: ABCDEF123456
Join code:   XY2P9Q
```

If `DISCORD_WEBHOOK_URL` is unset or the POST fails, codes fall back to stdout. No hard dependency on Discord — the game runs without it.

## Code Specs

- **Host secret**: 12-char random string, same charset as today's join code (`ABCDEFGHJKMNPQRSTUVWXYZ23456789`)
- **Join code**: 6-char, same format as today — pre-generated by `SessionCodeManager` rather than inside `GameSession`

## Files to Create / Modify

| File | Change |
|---|---|
| `server/SessionCodeManager.js` | **New.** Owns both codes, rotation logic, Discord posting, midnight timer |
| `server/discord.js` | **New.** Single `postCodes(hostSecret, joinCode, reason)` function; HTTP POST to webhook |
| `server/game/GameManager.js` | Instantiate `SessionCodeManager`; pass pre-generated join code into `getOrCreateSession()` |
| `server/game/GameSession.js` | Accept `joinCode` constructor param instead of self-generating; add `updateJoinCode()` for live rotation |
| `server/socket/handlers.js` | `lobby:setName`: check host secret before join code; `requireLanHost()`: also accept host-secret-authed players |
| `server/index.js` | Initialize `SessionCodeManager` on startup |

## New Environment Variable

| Variable | Default | Description |
|---|---|---|
| `DISCORD_WEBHOOK_URL` | *(not set)* | If set, sends host secret + join code to this webhook on every rotation. Falls back to stdout if unset or POST fails. |

## Security Notes

- Host secret is 12 chars (vs 6 for join code) to reduce guessability
- Discord webhook URL is sensitive and must be kept in env — never logged or broadcast
- Host secret and join code never appear in any client-side socket payload
- If webhook fails, codes fall back to stdout (SSH fallback)
- LAN realm path is unchanged — LAN players always bypass code checks

## Verification Checklist

1. Start server without `DISCORD_WEBHOOK_URL` → codes appear in stdout
2. Set `DISCORD_WEBHOOK_URL` → Discord message appears on startup with both codes
3. Join with host secret → becomes host, can start game, sees join code in UI
4. Join with join code → regular player, cannot start/reset
5. Join with wrong code → rejected with error
6. Trigger `lobby:reset` → new codes sent to Discord; old codes rejected
7. Confirm LAN-realm join still works with no code (existing behaviour unchanged)
8. Mock or wait for midnight → rotation fires, new codes sent to Discord
